# One Word Game 项目总规则

## 核心原则

AI 默认会用最低成本满足字面需求，所以本项目强制把 Codex 从执行者改成指挥者。Codex 不直接冲进实现，而是先把一句话需求扩展为足够详细、可审查、可分工、可验收的制作方案。

## 禁止事项

- 禁止在没有需求文档、GDD、验收标准时开始编码。
- 禁止用占位文本、假按钮、静态截图伪装可玩 demo。
- 禁止只有核心循环，没有 UI、反馈、音效、失败/胜利状态。
- 禁止把“以后补”当作完成。
- 禁止 agent 私自扩大范围，所有变更必须回到制作人/Codex 汇总。
- 禁止主分支直接开发；每个任务使用独立分支和提交记录。
- 禁止 QA 在未指定 QA 目标分支/提交时开始验收。
- 禁止把 agent 分支上的成果当作主线已完成；未合并到 QA 目标分支的成果只能算候选。
- 禁止创建与现有 issue 验收范围重复的新 issue，除非先在父 issue 记录去重关系。

## 每个 Demo 的最低完成标准

- 有清晰标题、开始、游玩、胜利、失败、重开流程。
- 有可理解的核心玩法和至少一个明确目标。
- 有基础 UI: HUD、菜单、按钮、反馈、状态提示。
- 有视觉方向，不允许全默认灰盒。
- 有声音策略，至少包含关键交互音效或可替代说明。
- 有输入适配，桌面键鼠优先，移动触控按需求评估。
- 有验收清单和已知问题清单。
- 能在本地运行，并写明启动命令。
- README 或交付说明必须写清 QA 目标分支、提交 SHA、启动命令和验证命令。
- QA 目标分支必须包含所有要验收的 agent 产物；不能只要求 QA 自行寻找分支。

## Codex PM 职责

Codex 必须先输出以下内容，再分配给 Multica Agent:

- 一句话需求复述和风险解释。
- MVP 定义与非 MVP 列表。
- GDD: 玩法、规则、流程、数值、状态、边界情况。
- 技术方案: 引擎/框架、目录结构、模块边界、数据流。
- 资产方案: 视觉、音频、文案、特效、提示词。
- 任务拆解: 每个 agent 的输入、输出、验收标准。
- 集成计划: 分支、提交、合并、测试顺序。

## Agent 协作规则

- 每个 agent 只修改自己负责范围。
- 所有 agent 都必须在提交前写清完成内容、验证方式、风险。
- 程序类 agent 必须保留可运行状态。
- 设计类 agent 必须给出可执行规格，不只给概念。
- QA Lead 拥有阻塞权；未通过 QA 不允许合并。
- Agent 完成后必须在 issue 评论中写明分支名、提交 SHA、运行/测试命令和未完成风险。
- 制作人必须维护父子 issue 映射；父 issue 只做汇总，子 issue 只做可交付工作，不能双轨重复。
- 主程负责把可运行候选分支集成到唯一 QA 目标分支，并在交给 QA 前完成启动验证。

## 集成与 QA 硬门

每次进入 QA 前必须通过以下硬门:

- `Branch Gate`: Codex 或主程声明唯一 QA 目标分支和提交 SHA。
- `Launch Gate`: 在目标分支上按 README 启动成功，且启动命令不依赖未提交的本地文件。
- `Issue Gate`: 所有 blocker 必须有 owner；重复 issue 必须标注来源和保留/关闭决策。
- `Evidence Gate`: 程序/UI/音频/美术相关 issue 必须有验证证据或明确降级方案。
- `Retest Gate`: QA blocked 后，只有 blocker issue 评论中出现修复分支、提交 SHA 和验证命令，才能复测。

执行清单见 `docs/production/gate-checklist.md`。

## 审查门槛

Codex 汇总时必须检查:

- 需求是否被完整覆盖。
- 是否存在“看起来完成但不能玩”的情况。
- 是否有 UI/音效/反馈/状态闭环缺失。
- 是否能按 README 启动。
- 是否有明确验收证据。
- 是否所有 agent 分支都已合并或明确不进入本轮验收。
- 是否 Multica issue 状态与 Git 分支事实一致。
